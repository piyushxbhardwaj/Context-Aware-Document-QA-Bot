SYSTEM ARCHITECTURE OVERVIEW
=============================================================================

Nexus is composed of five independently scalable planes:

  ┌─────────────────────────────────────────────────────────────────────┐
  │  CLIENT LAYER                                                       │
  │  Web Dashboard (React)  │  CLI (nexus-cli, Go)  │  REST / gRPC API │
  └──────────────────┬──────────────────────────────────────────────────┘
                     │
  ┌──────────────────▼──────────────────────────────────────────────────┐
  │  CONTROL PLANE  (multi-tenant, hosted by Meridian Labs in CLOUD)   │
  │  ┌─────────────┐ ┌───────────────┐ ┌──────────────┐ ┌───────────┐ │
  │  │  API Gateway│ │ Identity Svc  │ │ Pipeline Ctrl│ │ Scheduler │ │
  │  └─────────────┘ └───────────────┘ └──────────────┘ └───────────┘ │
  │  ┌─────────────┐ ┌───────────────┐ ┌──────────────┐               │
  │  │  Model Reg. │ │ Monitor Ctrl  │ │ Billing Svc  │               │
  │  └─────────────┘ └───────────────┘ └──────────────┘               │
  └──────────────────┬──────────────────────────────────────────────────┘
                     │  mTLS gRPC
  ┌──────────────────▼──────────────────────────────────────────────────┐
  │  DATA PLANE  (runs in CUSTOMER's cloud account — BYOC)             │
  │  ┌──────────────────────┐  ┌─────────────────────────────────────┐ │
  │  │  Pipeline Executor   │  │  NexusServe (KServe + Triton)       │ │
  │  │  (Kubernetes Jobs)   │  │  ┌──────────────────────────────┐  │ │
  │  └──────────────────────┘  │  │  Inference Pods (GPU / CPU)  │  │ │
  │  ┌──────────────────────┐  │  └──────────────────────────────┘  │ │
  │  │  Artifact Store      │  │  ┌──────────────────────────────┐  │ │
  │  │  (S3 / GCS / Blob)   │  │  │  Redis Response Cache         │  │ │
  │  └──────────────────────┘  │  └──────────────────────────────┘  │ │
  │  ┌──────────────────────┐  └─────────────────────────────────────┘ │
  │  │  NexusData (Delta)   │                                          │
  │  └──────────────────────┘                                          │
  └─────────────────────────────────────────────────────────────────────┘

The separation between control and data planes is the core security
architectural decision: customer data (training data, model weights,
inference payloads) never leaves the customer's cloud account.
The control plane only transmits orchestration metadata (run status,
metric scalars, logs, drift scores) — never raw data.

=============================================================================
PIPELINE EXECUTION ENGINE
=============================================================================

When a run is triggered:

1. Pipeline Controller validates the spec, resolves artifact references, and
   creates a Run record in the control-plane PostgreSQL database.

2. Scheduler assigns the run to a target data-plane namespace and emits a
   RunSpec message to a Kafka topic (nexus.runs.pending).

3. Data-plane Pipeline Agent (a long-running Kubernetes Deployment) consumes
   the RunSpec, translates each step to a Kubernetes Job (or Argo Workflow
   Task in Enterprise installs), and submits them respecting DAG dependencies.

4. Each step container writes output artifacts to the Artifact Store (signed
   S3 PUT URLs pre-generated by the control plane). On completion the step
   POSTs a StepDoneEvent back to the control plane via the Data Plane Proxy.

5. The Pipeline Controller transitions the run status and emits webhook events.

Step Isolation:
   - Steps run in separate Kubernetes namespaces by default.
   - Secrets (env vars, volume mounts) are injected via Kubernetes Secrets,
     never embedded in the RunSpec.
   - Network policy restricts inter-step communication to artifact store only.

Retry Policy:
   - Steps support up to 3 automatic retries on transient failure (OOM,
     node preemption). Retry count and backoff are configurable per step.
   - Idempotency: steps that write to the Artifact Store use content-addressed
     paths (SHA-256 of output), so retries don't duplicate artifacts.

=============================================================================
NEXUSSERVE INTERNALS
=============================================================================

Model Loading:
   When a deployment is created, NexusServe Controller:
   1. Pulls the model artifact from the Artifact Store to a shared PVC.
   2. Selects the appropriate serving runtime (ONNX Runtime, Triton, custom
      Python server) based on the model's framework tag.
   3. Creates a KServe InferenceService with the resolved image + config.
   4. Waits for the pod readiness probe (HTTP GET /v2/health/ready) before
      marking the deployment as "ready" and activating the endpoint URL.

Traffic Splitting:
   Canary deployments are implemented using Istio VirtualService weights.
   A canary_weight=20 means 20% of incoming requests are routed to the new
   deployment, 80% to the primary. Weights are updated live via Istio API
   without pod restarts.

Response Caching:
   The serving gateway (Envoy) checks a Redis cache keyed by BLAKE2b(request_body).
   Cache TTL is set per deployment (default 0, i.e., disabled). Cache hit rate
   is exposed as a Prometheus metric: nexus_serving_cache_hit_ratio.

Autoscaling:
   KEDA ScaledObject monitors two metrics:
   - requests-per-second (from Istio telemetry)
   - GPU memory utilization (from DCGM exporter)
   Scale-up is triggered when RPS exceeds (target_rps × current_replicas).
   Scale-down has a 5-minute cooldown to prevent thrashing.

=============================================================================
DATA PLANE PROXY
=============================================================================

The Data Plane Proxy is a lightweight Go service deployed in the customer's
cluster. It acts as the outbound-only bridge between the data plane and the
control plane:

  - Authenticates to the control plane via a workspace-scoped mTLS certificate
    (rotated every 24 hours by the control plane's Certificate Authority).
  - Forwards step completion events, log chunks (compressed), and metric
    scalars to the control plane via gRPC streaming.
  - Does NOT accept inbound connections from the control plane — all
    communication is customer-initiated.

This design means no inbound firewall rules are required in the customer VPC.
The only required egress rule is: TCP 443 to api.nexus.meridian-labs.io.

=============================================================================
NEXUSDATA — DATASET VERSIONING
=============================================================================

NexusData manages dataset versions using Delta Lake on top of the customer's
object store. Key concepts:

  Dataset    : a named, schema-versioned collection of records
  Snapshot   : an immutable point-in-time view of a dataset (Delta version)
  Lineage    : directed graph linking datasets → pipeline runs → model versions

Dataset operations are transactional (ACID): concurrent writers use
optimistic concurrency control with Delta Lake's transaction log.

Schema evolution is supported for:
  - Adding new nullable columns (backward compatible)
  - Renaming columns via schema migration scripts

Breaking changes (dropping columns, changing types) require creating a new
dataset version with an explicit migration.

Data profiles (row count, null rate, min/max/mean per numeric column) are
computed automatically on snapshot creation and stored alongside the snapshot
metadata.

OBSERVABILITY
=============================================================================

All Nexus components emit:
  - Structured JSON logs to stdout (captured by customer's log aggregator)
  - Prometheus metrics scraped by the customer's Prometheus instance
  - OpenTelemetry traces with W3C trace context propagation

Key metrics exposed by each component:

  Pipeline Agent:
    nexus_pipeline_run_duration_seconds{status}
    nexus_pipeline_step_attempts_total{step_name, status}
    nexus_artifact_upload_bytes_total

  NexusServe Gateway:
    nexus_serving_requests_total{model, version, status_code}
    nexus_serving_latency_seconds{model, version, quantile}
    nexus_serving_cache_hit_ratio{deployment_id}

  Data Plane Proxy:
    nexus_proxy_events_forwarded_total
    nexus_proxy_grpc_errors_total

Grafana dashboard templates for all components are provided in the
installation Helm chart under grafana-dashboards/.

=============================================================================
HIGH AVAILABILITY & DISASTER RECOVERY
=============================================================================

Control Plane (Nexus Cloud):
  - Deployed across 3 Availability Zones in the hosting region
  - PostgreSQL (Aurora): Multi-AZ with automated failover (< 30 s RTO)
  - Kafka: 3-broker cluster with replication factor 3
  - RTO: 30 minutes (declared SLA for full region failure)
  - RPO: 1 hour (PostgreSQL point-in-time recovery, 5-minute WAL shipping)

Data Plane (Customer's cluster):
  - Stateless components (Pipeline Agent, Data Plane Proxy): Kubernetes
    Deployment with 2 replicas; pod disruption budget prevents both from
    being evicted simultaneously.
  - Artifact Store: durability delegated to the underlying object store
    (S3 / GCS 11-nine durability).
  - NexusServe pods: min_replicas defaults to 2 in Enterprise installs.

=============================================================================
SECURITY HARDENING NOTES
=============================================================================

- Container images are built from distroless base images and scanned via
  Trivy on every release. Critical CVEs block the release pipeline.
- All inter-service communication is mTLS. Plain HTTP is rejected at the
  ingress with HTTP 301.
- Secrets in the control plane are encrypted at rest using AWS KMS (Cloud)
  or a customer-provided KMS key (Enterprise with BYOK).
- PII fields in logs (email, IP) are automatically redacted by the log
  pipeline using a configurable regex allowlist.
- Nexus undergoes annual penetration testing by a third-party firm.
  Findings and remediation status are shared with Enterprise customers
  under NDA via the security portal.
